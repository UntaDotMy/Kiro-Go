Web Search
Services an inbound web_search tool (Claude hosted web_search / Claude Code WebSearch). Default on. With Kiro selected below, search runs on Kiro's native MCP endpoint (no key, billed to the Kiro account). Pick an external provider to search via a standalone API instead — useful for a pure DashScope/Gemini/OpenAI deployment with no Kiro account.
Format used to surface chain-of-thought when streaming through the OpenAI-compatible endpoint. The label refers to the field/tag convention each ecosystem popularized — it is not a model selector.
Per research (jwadow/kiro-gateway #58): the codewhisperer.<region> hostname only resolves in us-east-1; outside us-east-1 we substitute runtime.<region>.kiro.dev. Q Developer (q.<region>) is verified working in us-east-1, eu-west-1, eu-central-1. Other regions may work but are unverified. Overridden by KIRO_API_REGION env.
Comma-separated AWS regions. When set, every endpoint in the first region is tried before any endpoint in the second, and so on — extra mitigation when the primary region throttles your entire pool. Empty = single-region (Primary above). Overridden by KIRO_API_REGIONS env. No competitor proxy does this; it is novel and unverified at scale, so use deliberately.
How the pool picks the next account. Fast (default) spreads requests in parallel across the least-loaded free accounts (WiFi-7 multi-link): each account holds up to its Concurrency/account cap (default 1 = send-and-ack), so a burst of N requests lands on N distinct accounts at once instead of queueing on one. When every account is at its cap the request waits for one to free and routes there — nothing is dropped. It does NOT apply a timed cooldown or adaptive concurrency limit; an in-use account simply isn't picked until it frees. Least-request steers each request to the least-busy account and applies a per-account adaptive concurrency limit (grows on success, halves on a 429) plus the soft cooldown — better when you want backoff under sustained heavy load. SWR is balanced rate-based rotation but blind to concurrency. Least-used tilts traffic toward less-burned accounts. Random is a control.
Fast strategy only. How many requests a single account handles at once before the next request fans out to another free account. 1 (default) = strict send-and-ack: one request per account at a time, so N concurrent requests spread across N accounts in parallel. When every account is at this cap the request waits for one to free (woken the instant it does). Raise it to let each account pipeline multiple requests. Empty/0 = default (1).
Least-request only. How many parallel requests a fresh or just-recovered account starts admitting (the "initial window"). Higher = faster ramp-up, more like unthrottled round-robin; lower = gentler on 429-sensitive accounts. Empty/0 = default (12). The limit grows fast (slow-start) toward the ceiling below, then probes gently above the point where the account last hit a 429.
Least-request only. The ceiling the adaptive limit can climb to as an account proves it can absorb load without throttling. Empty/0 = default (48). Must be ≥ initial concurrency. Caps how much of a burst any single account (AWS identity) can soak up before the pool spreads to others.
